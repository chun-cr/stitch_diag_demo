# Session 030 - 认证接口文档对齐修复

## 做了什么
- 按 `docs/接口.md` 与现有验证码联调契约复查认证链路，修正与后端文档不一致的请求字段、退出登录行为和测试断言。
- 将密码登录请求体收敛为文档约定的 `loginValue / countryCode / password / inviteTicket`，移除了遗留的 `phoneNumber` 与空字段透传。
- 为 `AuthInterceptor` 补上 401 后基于 `/api/v1/saas/mobile/auth/tokens/refresh` 的自动换发与重试逻辑，并新增回归测试覆盖。
- 将 Profile 页退出登录改为 best-effort 调用 `/api/v1/saas/mobile/auth/logout`，再清理本地 session 并回到登录页。
- 将注册验证码发送链路改回严格的 `challenge -> send` 时序，不再额外假设 challenge 会直接返回 send 结果。

## 变更前快照
- 密码登录请求体仍沿用旧的 `phoneNumber` 字段，并且会携带文档未定义的空字段。
- 密码登录未透传 `inviteTicket`。
- 个人页退出登录只清本地 token，没有调用后端 `/api/v1/saas/mobile/auth/logout`。
- 注册验证码发送链路保留了“challenge 直接返回 send 结果即可跳过 send 接口”的兼容分支。

## 问题与决策
- 决策：密码登录不再复用 `AuthRequest.toJson()`，而是按文档手工构造 payload，避免继续发出未定义字段和空值。
- 决策：`inviteTicket` 在密码登录与验证码认证时均透传，但仍只在文档允许的登录/认证终点上传，不影响 challenge/send/captcha 三步。
- 决策：退出登录采用 best-effort 方式先调后端 `/auth/logout`，即使服务端注销失败也继续清理本地状态，避免客户端卡死在“半退出”状态。
- 决策：刷新令牌换发放在 `AuthInterceptor` 内部统一处理，命中 401 时自动刷新并重试一次，避免把刷新逻辑散落到各业务页。
- 决策：注册验证码链路统一回到文档主流程，只信任 `challengeId` 与 captcha 信息，不再消费 challenge 中可能夹带的 send 结果扩展字段。
