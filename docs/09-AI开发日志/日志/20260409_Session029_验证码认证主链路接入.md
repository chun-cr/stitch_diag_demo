# Session 029 - 验证码认证主链路接入

## 做了什么
- 按联调文档接入移动端手机号验证码认证主链路：登录页与注册页均开始使用 `challenge -> send -> authenticate`。
- 新增 verification-code 的 challenge/send 数据模型与 repository 接口，页面端保存 `challengeId`、`expireAt`、`resendAt` 等必要状态。
- 登录页默认验证码主路径接到真实验证码认证流程，同时保留密码登录作为次入口。
- 注册页账号创建接到 `REGISTER` 场景验证码认证流程，不再依赖密码注册主流程。
- 对手机号/区号变化、challenge 失效、重发时序等前端状态做了重置与保护。
- 验证码倒计时已改成以后端 `resendAt` 为准，而不是固定 60 秒常量。
- 当 `challenge` 已存在但手机号/区号变化时，前端会立即失效旧 challenge，避免验证码与新号码脱钩。
- 登录页与注册页都对 `11119` / `11121` 做了 challenge 失效重置处理。
- 保留密码登录次入口，同时验证码链路已经成为登录页主路径。

## 变更前快照
- 登录页已有验证码 UI 与发送按钮三态，但 `sendCode()` 仍为前端 stub。
- 登录页真实后端已接的是手机号密码登录，不是验证码 authenticate。
- 注册页 UI 已是手机号 + 验证码，但创建账号提交仍错误走密码注册接口。
- 代码里还没有 `challengeId`、`captchaRequired`、`resendAt`、`expireAt` 等验证码流程状态模型。
- `sendCode()` 旧接口还错误地按 `countryCode + phoneNumber` 直发，不符合 challenge 绑定的文档约束。

## 问题与决策
- 决策：本轮以验证码认证主链路为主，登录页保持“验证码为主、密码为辅”的双模式结构。
- 决策：注册页按用户确认，正式切换成“手机号 + 验证码”主路径，不再以密码作为建号主流程。
- 决策：对于 `captchaRequired=true` 的 challenge，本轮如果没有明确 SDK 集成方式，则先做清晰的前端阻断和错误提示，不在缺契约状态下假实现。
- 决策：验证码倒计时将改成以后端 `resendAt` 为准，不再固定写死 60 秒。
- 决策：继续保留密码登录作为次入口，不在本轮验证码接入里移除。
- 决策：`sendCode()` 不再接受手机号直发，而是严格依赖 `challengeId`，以匹配文档主链路。
- 决策：当手机号或区号变化时，立即清空当前 challenge 与验证码倒计时状态，防止验证码与新输入脱钩。
- 决策：当服务端返回 `11119` / `11121` 时，直接视为当前 challenge 失效并重置本地验证码状态。
- 决策：当前 `captchaRequired=true` 仍以明确错误提示阻断，因为尚无 provider SDK 集成方式；不在缺 SDK 的情况下假实现 captcha/verify。
- 决策：保持密码注册接口仍可用，但注册页面主链路已切换到验证码 authenticate，不再依赖 `/register/password` 完成建号。

## 遗留风险
- captcha provider SDK 接入方式未明确，是本轮最大的外部不确定项。
- 当前对 `captchaRequired=true` 的处理是前端阻断并提示“当前环境暂不支持人机验证”，这符合当前缺 SDK 现状，但还不是真正完整能力。
- `X-App-Id` 目前在 `DioClient` 中仍使用默认占位值 `stitch_diag_demo`；若联调环境要求正式 AppId，仍需替换为后端认可值。
- 如果后端后续要求验证码登录完全替代密码登录，当前保留的密码模式需要再收一次范围。
- 注册页虽然已经切到验证码主链路，但“资料完善页数据提交后端”仍未在本轮实现；当前只保证认证与路由链路正确。
