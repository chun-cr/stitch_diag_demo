# Session 031 - 阿里云人机验证接入

## 做了什么
- 基于 `docs/20260410_app_flutter_aliyun_captcha_contract.md` 梳理 Flutter 验证码认证链路，把现有手工 `CaptchaResolver` 升级为阿里云 WebView/H5 人机验证方案。
- 保持登录页、注册页现有 `challenge -> captcha/verify -> send -> authenticate` 主链路不变，仅在 resolver 层新增 `ALIYUN` provider 分支。
- 为 resolver 接口补充 `challengeId` 透传，新增阿里云 payload 解析、H5 URL 构造、JS bridge 消息解析与 `AliyunCaptchaPage` WebView 容器页。
- 将 resolver 与页面中的新增用户可见文案接入现有 `gen-l10n` 体系，并新增 focused tests 覆盖阿里云 payload / bridge 解析逻辑。

## 变更前快照
- 登录页与注册页已经接入 `challenge -> captcha/verify -> send -> authenticate` 主链路。
- 当前 `captchaResolverProvider` 默认返回 `ManualCaptchaResolver`，需要用户手工粘贴 provider 返回的 JSON 结果。
- 仓库已引入 `webview_flutter`，但 auth 流程里尚未落地实际 WebView 人机验证页面。

## 问题与决策
- 决策：保持登录/注册页现有业务主流程不变，只替换 captcha resolver 的实现方式。
- 决策：`ALIYUN` provider 走 WebView + H5 bridge，其他 provider 先保留手工 resolver 兜底，避免扩大影响范围。
- 决策：为便于 H5 容器页桥接与校验，resolver 接口增加 `challengeId` 参数，并由页面调用方透传。
- 决策：`captchaVerifyParam` 在 Flutter 侧只做“非空字符串”校验，不做 trim、拆分或重编码，保持对后端透传原值。
- 决策：阿里云 payload 要求 `sceneId / prefix / userCertifyId / h5Url` 完整且 `h5Url` 为 HTTPS，否则视为当前环境未正确配置并中断流程。
