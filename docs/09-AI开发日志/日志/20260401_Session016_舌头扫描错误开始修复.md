# Session 016 - 舌头扫描错误开始修复

## 做了什么
- 启动排查舌头扫描在未真正检测到舌头时也能开始扫描的问题，覆盖 Flutter 页面、Dart bridge、Android / iOS 原生舌头判定与现有回归测试。
- 收敛最小修复范围：优先修正 `TongueScanStatus.readyToScan` 的 bridge 语义，避免 `TongueScanPage` 在 fallback / 低分状态下提前开始 hold tracking，不先扩大到原生阈值改动。
- 完成最小实现：将 `TongueScanStatus.readyToScan` 从 OR 判定收紧为 AND 判定，要求 `mouthPresent && tongueDetected && tongueOutScore >= 0.26` 才允许进入扫描就绪态。
- 清理测试文件中的冗余 `material.dart` import，保证 changed scope `flutter analyze` 无额外告警。

## 变更前快照
- `tongue_scan_page.dart` 在 `_handleStatusUpdate()` 中直接以 `status.readyToScan` 驱动 `_tongueDetected` 和 `_startHoldTracking()`。
- `tongue_scan_status_bridge.dart` 当前把 `readyToScan` 定义为 `mouthPresent && (tongueDetected || tongueOutScore >= threshold)`，只要原生 `tongueDetected` 为真就可能提前放行。
- 现有测试 `test/features/scan/tongue_scan_status_bridge_test.dart` 已明确期望：低 `tongueOutScore` 场景下 `readyToScan` 必须为 `false`。

## 问题与决策
- 决策：先用已有 bridge 回归测试做红灯验证，确认 bug 已被现有测试覆盖，再做最小实现修复。
- 决策：本次不新增文案，不改页面布局与原生通道契约。
- 决策：即便 Android / iOS 原生当前仍保留较宽松 fallback，本次也先通过 Dart bridge 恢复严格就绪条件，确保跨端行为立即收敛。
