# Session 041 - 面部近脸上传门槛调整

## 做了什么
- 调查“脸离得近时，即使界面显示面部已就位，也无法触发上传”的原因。
- 确认当前面部扫描存在 UI 提示与真实上传门槛不一致：椭圆框状态气泡只根据 `_hasFaceDetected` 与 `_faceDirection` 显示“已就位”，而自动开始上传还要求 strict framing。
- 已完成两类调整：
  1. 让“已就位”提示与真实 strict hold eligibility 对齐；
  2. 适度放宽 strict near-face framing 阈值，让中度近脸可触发上传，但保留“太近”拦截。
- 根据后续需求变更，已将 UI 提示再次改回原来的检测判断逻辑：只要检测到脸且无方向纠偏提示，椭圆框状态气泡仍显示“已就位”；同时保留近脸 strict 阈值放宽方案，让用户无需后退过多也能进入上传。

## 变更前快照
- `face_scan_page.dart` 的 `shouldAutoStartFaceScan()` / `shouldBeginFaceScan()` 需要 `isFramed=true`。
- `_buildOvalFrame()` 中的状态气泡在 `_hasFaceDetected=true && _faceDirection.isEmpty` 时就会显示 `scanFaceDetectedReady`，即使 `isFramed=false`。
- strict framing 当前阈值：`minArea=0.04`、`maxArea=0.46`、`guideInsetFactor=0.04`；近脸最容易被 `too_large` / `framing_failed` 拦截。

## 问题与决策
- 决策：先补纯逻辑测试，不直接上 widget test，避免把复杂 PlatformView 与状态流引进来。
- 决策：严格区分“检测到脸”和“允许开始上传”；修复后 UI 的 ready 提示只在 strict eligibility 满足时出现。
- 决策：strict 阈值只做小幅放宽，避免把“明显太近”的场景也放进来。
- 实施：新增 `shouldShowFaceReadyStatus(...)`，并让椭圆框状态气泡的 ready 文案与高亮样式都改为使用 strict hold eligibility。
- 实施：新增 `isFaceFramedForUploadBounds(...)` 纯逻辑 helper，把 strict 阈值从 `maxArea=0.46 / guideInset=0.04` 调整为 `maxArea=0.50 / guideInset=0.02`；relaxed 阈值保持 `maxArea=0.50 / guideInset=0.02`。
- 实施：同步更新 `Face hold diagnostics` 里的 `too_small / too_large / framing_failed` 判定，避免日志仍输出旧阈值。
- 回退：`shouldShowFaceReadyStatus(...)` 最终改回基于 `hasFaceDetected + faceDirection.isEmpty` 的原始判断，只回退 UI 提示，不回退 near-face 阈值优化。
- 三次调整：根据 Android 真机反馈“仍需后退很多”，继续沿 `maxArea / inset` 单线放宽。最终阈值为：
  - strict：`maxArea=0.52`、`guideInsetFactor=0.0`
  - relaxed：`maxArea=0.54`、`guideInsetFactor=0.0`
  这次调优的直接依据是 fail-first 测试证明：仅把 strict `maxArea` 提到 `0.52` 仍不足以放过“贴满提示框边缘”的近脸场景，说明 `inset` 同样是实际拦截点。

## 下次计划

## 验证结果
- Fail-first：`flutter test test/features/scan/face_scan_hold_test.dart -r compact` 在补 helper 前失败，明确提示缺少 `shouldShowFaceReadyStatus` 与 `isFaceFramedForUploadBounds`。
- 二次 fail-first：在将 UI 提示测试改回“检测到脸即显示 ready”后，同一测试文件再次失败，明确提示 `shouldShowFaceReadyStatus` 的命名参数和语义仍停留在 strict-ready 版本。
- 三次 fail-first：
  - `isFaceFramedForUploadBounds accepts a slightly closer face under strict framing` 在仅调 `maxArea=0.52` 前失败，证明 strict 上限仍过紧。
  - `isFaceFramedForUploadBounds keeps a slightly larger face alive under relaxed framing` 也失败，证明 relaxed 侧同样需要同步放宽，且 guide inset 是实际拦截来源之一。
- 纯逻辑回归：`flutter test test/features/scan/face_scan_hold_test.dart -r compact` 通过，新增覆盖：
  - `shows ready once a face is detected and no directional correction is needed`
  - `does not show ready before a face is detected`
  - `accepts a moderately close face under strict framing`
  - `accepts a slightly closer face under strict framing`
  - `still rejects an obviously too-close face under strict framing`
  - `keeps a slightly larger face alive under relaxed framing`
  - `still rejects an obviously too-close face under relaxed framing`
- 诊断：`lsp_diagnostics` 对 `face_scan_page.dart` 与 `face_scan_hold_test.dart` 均无问题。
- `flutter analyze lib/features/scan/presentation/pages/face_scan_page.dart test/features/scan/face_scan_hold_test.dart`：通过。
- `flutter test test/features/scan -r compact`：通过，扫描模块回归全绿。

## 遗留风险
- 本次主要基于纯逻辑测试与扫描模块回归，没有 Android/iOS 真机实机验证；不同设备上近脸 landmarks 分布仍可能有差异。
- relaxed 阈值与 strict 阈值现在在“太大上限”和“guide inset”上已对齐，如果后续需要进一步区分“开始前更严、开始后更松”，需要基于真机数据重新拉开两组阈值，而不是继续拍脑袋微调。
- 当前 UI 提示已按你的要求恢复为“检测到脸即可提示已就位”，因此界面语义再次与真实上传门槛分离；若未来继续出现“显示已就位但未上传”的反馈，应优先用 `Face hold diagnostics` 日志解释 `too_large / framing_failed / accepted_snapshot_missing`，而不是仅依赖提示文案判断。
