# Session 042 - 面诊描点图上传保护

## 做了什么
- 启动面诊上传保护修复，目标是确保 `faceFrameFile` 始终代表带面部描点的人脸图，而不是在 landmarks 缺失时退回原始裁剪图。
- 采用最小风险改动：在 Flutter 人脸提交流程里显式校验“描点是否可渲染 / 生成结果是否仍是原图”，不改后端契约，不扩散到手掌或舌诊链路。
- 在 `face_scan_page.dart` 新增 `hasRenderableFaceFrameUpload(...)` 纯逻辑 helper，并在上传前拦截“无 landmarks / 生成结果仍是原图”的 `faceFrameFile`。
- 新增 `face_scan_hold_test.dart` fail-first 逻辑测试，覆盖 helper 的三种语义：无点位拒绝、回退原图拒绝、生成独立 overlay 允许。
- 新增 `scanFaceFrameRetryMessage` 多语言文案，在阻止上传时通过 toast 明确提示用户重新扫描。

## 变更前快照
- `face_scan_page.dart` 会先基于 accepted snapshot 生成 `faceFrameUploadPath`，再把 `capture.croppedPath` 和 `faceFrameUploadPath` 一起上传。
- `renderFaceFrameMaskFile(...)` 在 landmarks 为空时会直接返回 `sourceImagePath`，这意味着 `faceFrameFile` 可能退回成无描点的原图。
- 当前项目已有 `face_scan_hold_test.dart` 这类纯逻辑测试，适合新增轻量 helper 测试，避免把 PlatformView / 相机桥接引入回归成本。

## 问题与决策
- 决策：优先在页面提交流程做保护，不修改 `renderFaceFrameMaskFile(...)` 的既有兜底语义，避免影响其他潜在调用方。
- 决策：先补 fail-first 纯逻辑测试，再补最小实现，保持 TDD 闭环。
- 决策：若检测不到可用于渲染描点图的 landmarks，则阻止上传并提示用户重新扫描。
- 实施：继续保留 `renderFaceFrameMaskFile(...)` 在空 landmarks 时返回原图的底层行为，但在页面上传前通过 `hasRenderableFaceFrameUpload(...)` 显式拦截，不让该回退结果流入后端接口。
- 实施：用户提示采用新的 `scanFaceFrameRetryMessage` 本地化 key，不复用泛化的“上传失败”弹窗，避免在业务上误导成网络失败。
