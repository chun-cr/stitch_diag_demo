# Apple Pay 无后端阶段实施清单（stitch_diag_demo）

## 1. 文档目标

本文档用于说明：在当前 **`stitch_diag_demo` 项目尚未具备后端支付能力** 的情况下，Apple Pay 可以先推进到哪一步、项目内应该先准备哪些结构、需要改哪些文件，以及未来后端接入时建议遵循的接口契约。

目标不是“先伪造支付成功”，而是：

1. 先完成 **Apple Developer / iOS 工程 / Flutter 客户端结构准备**
2. 保持当前 mock checkout 语义清晰
3. 让未来接真实 Apple Pay 时尽量少改页面层

---

## 2. 当前项目现状

基于当前仓库代码，Apple Pay 仍处于 **占位阶段**。

### 2.1 当前已存在的相关文件

- `lib/features/report/presentation/pages/report_checkout_page.dart`
  - 已有 Apple Pay 按钮入口
  - 当前点击只弹说明，不是真实支付
- `lib/features/report/application/mock_product_checkout.dart`
  - 当前只负责 mock 金额预览 / mock checkout 计算
- `lib/core/router/app_router.dart`
  - 已有 checkout 独立路由，适合后续局部升级
- `pubspec.yaml`
  - 当前只有 `in_app_purchase`
  - 没有 Apple Pay 专用 Flutter 依赖

### 2.2 当前缺失的关键项

- 项目内未发现 iOS Apple Pay entitlement 文件
- `ios/Runner.xcodeproj/project.pbxproj` 当前未接入 Apple Pay capability 配置
- 当前没有 Merchant ID / Apple Pay capability / payment processing certificate 的仓库接线痕迹
- 当前没有后端 checkout session / payment confirm 接口
- 当前没有真实 Apple Pay token 提交与确认链路

### 2.3 重要结论

当前项目不是“差最后一小步就能真实支付”，而是处于：

> **前端 mock checkout + Apple Pay 占位入口**

因此当前阶段最合理的目标应是：

> **先把 Apple Pay 做到“结构就绪、工程就绪、配置就绪”，而不是“先假装真实可支付”。**

## 3. 没有后端时，Apple Pay 可以做到哪一步

## 3.1 现在就能做的

1. Apple 开发者后台配置
2. iOS Target Apple Pay capability 接线
3. entitlement / merchant 配置接入
4. Flutter checkout / payment 抽象层设计
5. Apple Pay 客户端服务接口设计
6. mock checkout session 模型设计
7. 页面状态机从“mock 按钮”升级为“待接真实支付方式”

## 3.2 现在做不了的

1. 真实扣款
2. 后端验 Apple Pay token
3. 服务端创建可信支付订单
4. 对账、补单、退款、支付状态闭环
5. 可信“支付成功”结果

## 3.3 最好情况下能做到哪里

如果后续补上客户端插件或原生桥接，但仍没有后端，那么最多可以做到：

- 检测设备是否支持 Apple Pay
- 展示 Apple Pay 按钮
- 拉起 Apple Pay 支付面板原型
- 拿到客户端授权结果 / token 数据

但即使做到这里，也仍然不能把结果定义为：

> “真实支付成功”

因为订单是否成功，必须以后端确认为准。

## 4. Apple 开发者后台现在就应该做的事

这些工作不依赖你们后端代码，可以先完成。

## 4.1 确认正式发布主体

先确认以下三件事：

1. 最终发布使用哪个 Apple Developer Team
2. 最终 App 的 Bundle Identifier 是什么
3. 谁拥有创建 Merchant ID / capability / 证书的权限（通常是 Admin / Account Holder）

> 不建议继续以临时 `com.example...` 思路推进 Apple Pay 正式配置。

## 4.2 创建 Merchant ID

示例：

```text
merchant.com.yourcompany.stitchdiag
```

说明：

- Merchant ID 是 Apple Pay 的商户身份标识
- 它属于你们的 Apple Developer Team / 商户体系
- 它不是某个开发者个人电脑，也不是单独某个 Xcode 本地账号

## 4.3 给 App ID 开启 Apple Pay capability

将 App ID 与 Merchant ID 绑定。

## 4.4 创建 Payment Processing Certificate

这是和 Merchant ID 关联的支付处理证书，用于加密支付信息。

Apple 官方前置要求核心就是这三项：

1. Merchant Identifier
2. Apple Pay capability
3. Payment Processing Certificate

## 5. iOS 工程层实施清单

## 5.1 目标文件

### 当前已存在

- `ios/Runner/Info.plist`

### 预计需要新增 / 接线

- `ios/Runner/Runner.entitlements`（预计新增）

## 5.2 需要完成的工程动作

### A. 打开 Apple Pay capability

在 Xcode 中给 `Runner` target 打开 Apple Pay capability。

### B. 创建或接入 entitlement

预期会新增类似：

```xml
<key>com.apple.developer.in-app-payments</key>
<array>
  <string>merchant.com.yourcompany.stitchdiag</string>
</array>
```

### C. 在 project.pbxproj 中引用 entitlements

关键配置通常会出现类似：

```text
CODE_SIGN_ENTITLEMENTS = Runner/Runner.entitlements;
```

## 5.3 当前不需要重点动的文件

### `ios/Runner/Info.plist`

Apple Pay 本身不是靠大量 plist 配置完成的。当前阶段重点不在 plist，而在：

- capability
- entitlement
- merchant 绑定

所以：

> `Info.plist` 通常不是 Apple Pay 接入主战场。

## 8. 依赖与接入策略建议

## 8.1 当前依赖状态

### `pubspec.yaml`

当前已有：

- `in_app_purchase`

当前没有：

- Apple Pay 专用 Flutter 插件

## 8.2 建议策略

### 当前阶段

> 先做抽象层，不急着加 Apple Pay Flutter 插件。

原因：

1. 没有后端时，过早接插件会造成“前端能调起，但支付闭不了环”
2. 如果后续支付方案变更（例如接 Stripe/Adyen/其他 PSP），前端很容易返工

### 建议顺序

1. 先抽象 checkout / payment service / session model
2. 再决定 Apple Pay 的客户端接法：
   - 原生 PassKit bridge
   - Flutter `pay` 插件
   - PSP 官方方案

---

## 9. 需要和现有 IAP 严格分开的部分

当前项目已有：

- `lib/features/report/application/report_unlock_service.dart`
- iOS 本地 `StoreKit` 相关调试文件

这是 **In-App Purchase（IAP）** 路径，用于 App 内商品解锁。

Apple Pay 如果用于：

- 实物商品
- 服务商品
- 外部履约商品

就应该是**另一条完全独立的支付链路**。

> 不要把 `in_app_purchase` 当作 Apple Pay 的替代方案。

## 10. 后续后端接口草案

这一部分建议你现在就定下来，即使后端代码还没开始写。

---

## 10.1 创建 Checkout Session

### 接口

```http
POST /api/checkout/sessions
```

### 请求示例

```json
{
  "productId": "herbal-moxa-kit",
  "quantity": 1,
  "shippingAddress": {
    "recipient": "陈清和",
    "phone": "13800001234",
    "line1": "上海市徐汇区漕溪北路 88 号 18 楼"
  },
  "paymentMethod": "apple_pay"
}
```

### 返回示例

```json
{
  "sessionId": "chk_20260401_xxx",
  "items": [
    {
      "productId": "herbal-moxa-kit",
      "name": "艾灸调理礼盒",
      "quantity": 1,
      "unitPriceCents": 39900
    }
  ],
  "subtotalCents": 39900,
  "shippingFeeCents": 1200,
  "serviceFeeCents": 0,
  "totalCents": 41100,
  "currencyCode": "CNY",
  "countryCode": "CN",
  "supportedPaymentMethods": ["apple_pay"],
  "status": "pending"
}
```

### 作用

- 由后端返回权威金额
- 不再相信前端自己算的最终金额
- 作为后续支付确认的基准对象

---

## 10.2 确认 Apple Pay 支付

### 接口

```http
POST /api/payments/apple-pay/confirm
```

### 请求示例

```json
{
  "sessionId": "chk_20260401_xxx",
  "applePayToken": {
    "paymentData": "...",
    "transactionIdentifier": "...",
    "paymentMethod": {
      "network": "Visa",
      "displayName": "Visa 8888"
    }
  },
  "idempotencyKey": "8c1e0c5d-xxxx-xxxx-xxxx",
  "billingContact": {
    "email": "demo@example.com"
  },
  "shippingContact": {
    "recipient": "陈清和",
    "phone": "13800001234"
  }
}
```

### 返回示例

```json
{
  "paymentId": "pay_xxx",
  "orderId": "ord_xxx",
  "status": "authorized",
  "failureCode": null,
  "failureMessage": null
}
```

### 关键原则

Apple Pay 客户端授权成功 ≠ 订单支付成功。

必须以服务端确认结果为准。

---

## 10.3 查询最终状态

### 接口候选

```http
GET /api/checkout/sessions/{sessionId}
```

## 阶段 4：后端接入阶段

1. 创建 checkout session 接口

## 12. 对当前项目最实用的建议

对于 `stitch_diag_demo` 当前状态，最合理的策略不是“现在把 Apple Pay 直接接通”，而是：

> **先把 Apple Pay 从“占位按钮”升级成“结构完整、工程就绪、等待后端接入的正式支付方式”。**

这样做的收益是：

1. 当前 checkout 页面不会返工重写
2. 后续只需要替换 service / session 数据来源
3. Apple Developer / iOS 工程准备可以先完成
4. 产品、后端、客户端可以先围绕同一份接口契约工作

---

## 13. 一句话总结

### 现在可以完成的目标

**Apple Pay 配置准备 + iOS 接线准备 + Flutter 结构准备 + 后端 contract 设计**

### 现在不能承诺的目标

**真实支付完成与可信支付成功**

因为这一步必须依赖后端确认。
